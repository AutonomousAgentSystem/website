# Examples & Use Cases

Real-world scenarios showing AAS in action.

## Example 1: Automated Git Cleanup

**Scenario:** Your team commits frequently but forgets to cleanup branches.

**Setup:**

```json
{
  "agents": {
    "repository": {
      "enabled": true,
      "detection_interval": "60s",
      "confidence_threshold": 0.8
    }
  }
}
```

**What happens:**

1. **Hour 1** — AAS detects 10 stale branches
   - Analyzes which ones are merged: 2 LLM calls
   - Plans cleanup strategy
   - Deletes branches
   - Learns: "stale branch" → "delete if merged"

2. **Hour 2–10** — Same pattern detected 9 more times
   - AAS recognizes the pattern
   - Executes learned deletion strategy
   - **0 LLM calls per cycle** (100% cache hit)

**Cost savings:** 18 LLM calls → 2 LLM calls (90% reduction)

**Output:**

```
[INFO] repository: detecting issues...
[INFO] repository: found 10 stale branches
[INFO] repository: checking cache for solution...
[INFO] repository: cache miss - analyzing with LLM
[ANALYZE] Stale branches: main-old, feature-xyz, hotfix-v1.2, ...
[PLAN] Delete branches: [list]
[EXECUTE] Deleting 10 branches...
[VERIFY] Deleted successfully
[LEARN] Pattern: stale_branches → delete

---

[Second run, 1 hour later]

[INFO] repository: detecting issues...
[INFO] repository: found 8 stale branches
[INFO] repository: checking cache for solution...
[INFO] repository: cache HIT - reusing learned solution
[EXECUTE] Deleting 8 branches (0 LLM calls)...
[VERIFY] Deleted successfully
```

## Example 2: Automatic Log Rotation

**Scenario:** Your app logs grow unbounded and fill disk.

**Setup:**

```json
{
  "agents": {
    "logs": {
      "enabled": true,
      "detection_interval": "30s",
      "paths": ["/var/log/app.log", "/var/log/error.log"],
      "error_threshold": 50
    }
  }
}
```

**What happens:**

1. App produces 100 errors, log file hits 500GB
2. AAS detects: "log file too large, too many errors"
3. **First time:** Asks Claude: "How should I handle large error logs?"
   - Claude suggests: rotate, compress old logs, keep recent 5 days
   - AAS implements and learns this strategy
4. **Next time (week later):** Same issue
   - AAS executes learned strategy without LLM call
   - Saves $0.05–0.10 per rotation

**Cost per issue:** $0.01–0.03 (1–2 LLM calls) initially, $0 on repeats

- MTTR (Mean Time To Repair): 0s (no human intervention)

## Example 5: Learning Over Time (RSI)

**Scenario:** Agent success rate improves, becomes more aggressive.

**Initial config:**
```json
{
  "confidence_threshold": 0.7
}
```

**Over 100 cycles:**

```
Cycle 1–10:   Success rate 60% → confidence stays 0.7
Cycle 11–20:  Success rate 75% → confidence increases to 0.75
Cycle 21–30:  Success rate 82% → confidence increases to 0.80
Cycle 31–40:  Success rate 90% → confidence increases to 0.85
Cycle 41–50:  Success rate 95% → confidence increases to 0.90
```

**Effect:** Agent acts on lower-confidence signals as it proves itself.

**Before:** Waits for 70% confidence, misses issues
**After:** Acts at 90% confidence, catches issues faster (fewer false positives proven)

## Example 7: Confidence Threshold Tuning

**Scenario:** You want AAS to be more conservative (lower false positives).

**Initial state:**
```json
{
  "confidence_threshold": 0.7
}
```

Agent acts 70% of the time, sometimes breaks things.

**Adjust:**
```bash
aas config set agents.repository.confidence_threshold 0.95
```

**Result:**
- Acts only when 95% certain
- Fewer false positives ✓
- Slower response ✗
- Higher LLM cost (fewer pattern matches)

Trade-off: safety vs. speed.

## Example 9: Integration with Claude Code

**Scenario:** Code quality drifts, want AAS to auto-refactor.

**Setup:**

```json
{
  "integrations": {
    "claude_code": {
      "enabled": true
    }
  }
}
```

**Workflow:**

1. Repository agent detects code quality issue (e.g., "outdated dependency")
2. Claude Code agent analyzes the code
3. Plans refactor via LLM
4. Uses Claude Code CLI to apply changes
5. Verifies tests still pass
6. Commits and pushes

**Example action:**

```bash
# Claude Code refactors and commits
claude code-change -f src/main.rs \
  -c "Upgrade deprecated tokio APIs" \
  -a "apply"
```

```
  - Same pattern, cached solution, 0 LLM calls
  - Confidence increases from 0.7 → 0.95

Cycle 11 (Production):
  - Same pattern detected
  - Confidence: 0.95 (trusted from staging)
  - Execute: Run cleanup in production
  - Verify: Success
```

**Safety:** AAS learns in controlled environment, applies with confidence in production.

## Getting Started with Examples

1. Pick a scenario that matches your infrastructure
2. Copy the config from the example
3. Run in dry-run mode: `aas run --dry-run`
4. Verify plans look right
5. Enable: `aas daemon start`
6. Monitor: `aas logs follow`
7. Watch LLM costs drop as patterns are learned

Most sites see **80–95% LLM savings** after 2–4 weeks of operation.
